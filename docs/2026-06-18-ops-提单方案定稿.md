# ops 提单方案 · 定稿(去同步 + 工单快照 + 实时查 + 指派)

> 2026-06-18 · 合并并取代之前 3 份草稿。这是开发依据。

## 0. 一句话
**ops 不再同步 soyoo 数据。** 提单时实时查 soyoo;建单把用得上的字段**全部快照**进工单;看工单只查 ops 自己的库;soyoo 改信息时推一条让 ops 刷快照;换人不动旧工单。管理员看全部工单 + 可指派。

---

## 1. 数据归属

| | 内容 |
|---|---|
| **ops 自己存** | 工单(含全量快照)、环节 `ops_segments` + 标签绑定 `ops_segment_tags`、登录会话 |
| **soyoo 实时查(不落库)** | 项目、用户、客户、标签、项目成员/成员标签 |

**退役**:`people` / `projects` / `tenants` / `tags` / `project_team` / `project_member_tags` 六张同步表 + 同步调度 + 同步管理页(见 §7)。

---

## 2. 三条链路

### 2.1 提单(写,低频)
- 选 **客户 / 项目(我参与的)/ 负责人** → **实时查 soyoo**(新分组,见 §5)。
- 建单时**把能拿到的字段全部快照**进工单行(见 §3)。

### 2.2 看工单(读,高频)
- **只查 ops 自己的库**,工单快照里啥都有 → 刷新即出、**完全不碰 soyoo**。
- 列表/详情显示的 客户/项目/提单人(名+头像)/负责人(名+头像)/环节,全来自工单快照;环节名也可按 `segment_id` 取 ops 本地环节当前名。

### 2.3 soyoo 改了 → 推一条给 ops(分两种)
- **改信息**(同一实体改名/换头像:用户改名、项目改名、客户改名)→ soyoo 推 `{type,id}` → ops 按 id **更新对应旧工单的快照**(`UPDATE tickets SET owner_name=… WHERE owner_id=id` 等)。
- **换人 / 移除成员 / 删除**(项目成员 A→B、删项目/用户)→ **不动旧工单**(旧单还挂 A,A 做的留着)。只影响**新建提单能选谁**(选不到 A,只能选 B)—— 而"选谁"本就是实时查,天然正确。
- 结论:**改信息=刷旧工单快照;换人/删=不碰旧工单**。

## 3. 工单快照字段(建单时"能拿到的都存")

`tickets` 现有:`project_id`、`project_name`、`source_project_name`(客户名)、`owner_id`、`requester_id`、`segment_id`、`discipline`(环节名快照)、`tag_id`、title/priority/status/due_in_hours/content_html…

**新增快照列(把用得上的补全)**:
| 列 | 含义 |
|---|---|
| `client_id` | 客户(租户)id |
| `client_name` | 客户名(或复用 source_project_name) |
| `project_status` | 提单时项目状态 |
| `owner_id`(已有) `owner_name` `owner_avatar` `owner_username` | 负责人 |
| `requester_id`(已有) `requester_name` `requester_avatar` `requester_username` | 提单人 |
| `tag_name` | 环节对应标签名 |

> 原则:**建单时 soyoo 能返的、将来列表/详情/导出/筛选可能用到的字段都顺手存下来**(多存几列几乎零成本,省得以后补)。显示一律读这些快照。

## 4. 管理员 + 指派

- **工单列表可见性**:现在 `GET /api/ops/tickets` 只按 `owner_id/requester_id` 查"我的"。**加角色判断:管理员看全部,普通用户看自己的。**
- **指派/改派**:工单状态旁加「指派」。把工单转给**该项目的其他成员**(候选人 = 实时查 soyoo `/projects/:id/members`)。指派 = 更新 `owner_id` + 同步刷新 `owner_name/avatar` 快照。

## 5. soyoo 新增「对 ops 分组」(复用 soyoo 登录 token)

新分组(如 `/integration/*`)。**ops 登录本就走 soyoo、持有用户的 soyoo token**,所以用户级接口直接**透传该 token**,不另设共享密钥(详见 §11.3)。

| 接口 | 用途 | 现状 |
|---|---|---|
| `GET /users/:id/projects` | 提单"我参与的项目" | 已有 `/ops/users/:id/projects` |
| `GET /projects/:id/members` | 项目成员(选负责人/指派候选/算环节负责人) | 已有 |
| `GET /tenants` | 提单客户下拉 | 已有 |
| `GET /tags` | 环节配置绑定标签 | 已有 |
| `GET /projects/:id` | 单个项目(push 刷快照、建单校验) | **需新增** |
| `GET /users/:id` | 单个用户(push 刷快照) | **需新增** |

> **id 映射**:ops 历史 id 形如 `ops-user-{soyooId}`;调 soyoo 前还原成纯 soyoo id。建议工单新快照直接存**纯 soyoo id**,避免前缀来回转。
> **变更投递改 outbox(见 §11.1)**:soyoo 在「改用户信息 / 改项目名 / 改客户名」时**写一行 outbox**(不再 HTTP 推,挂了不丢);ops 按游标拉取消费。**换人/分配成员/删除不写**(不影响旧工单)。

## 8. 标签绑定(环节↔标签)会受什么影响?

**功能不变,数据源从本地表改成实时查 soyoo。**
- 绑定关系 `ops_segment_tags(segment_id, tag_id)` 在 **ops 自己库,不受影响**;tag_id 是 soyoo 标签 id,**稳定**。
- 环节配置页**选标签的列表** → 从 `GET /api/ops/tags`(本地)改成**实时查 soyoo `/tags`**。标签改名 → 配置页实时显示新名。
- **算环节负责人** → 从本地 `project_member_tags` 改成**实时查 soyoo `/projects/:id/members`**(返回成员+各自标签),service 里求"成员标签 ∩ 环节绑定标签"。
- **边缘情况**:某环节绑了一个**后来被删的标签** → 实时列表里没这个标签了,该环节算不出负责人 → 配置页提示"标签已失效,请重绑"。(同步方案也有这问题,不是新引入的。)

## 9. 落地顺序
1. **soyoo**:开带 token 的对‑ops 分组(补 `GET /projects/:id`、`/users/:id`);push 收窄到"改信息"。
2. **ops**:`soyoo-client` + service 分层;`tickets` 加快照列;建单实时查+写全量快照;列表管理员可见;指派接口+前端;环节配置/负责人改实时查;push 改"刷快照"。
3. **删**:全量同步 + 同步管理(§7)。
4. **验收**:改名→旧工单跟着变;换人→旧工单不动、新提单选不到旧人;管理员看全部 + 能指派;断开 soyoo 时看工单不受影响(提单暂不可用)。

## 11. 本轮确认补充(可靠性 / 表结构 / token / 标签)

### 11.1 变更投递:outbox + 游标拉(可靠不丢)
取代"HTTP 推一条"(ops 挂了就丢)。
- soyoo 加 append-only 表 `ops_change_outbox(seq BIGINT 自增主键, entity_type, entity_id, created_at)`;「改用户信息 / 改项目名 / 改客户名」时 INSERT 一行(尽量与业务写在同一事务)。
- soyoo 加 `GET /integration/changes?after=<seq>&limit=200`(返回 seq 之后的行)。
- ops 记 `last_seq`,每 ~30–60s 拉一次 → 逐条:查实体当前值 → UPDATE 对应工单快照 → **成功后**才推进 last_seq。
- **不丢**(ops 挂了 outbox 堆着,起来续传)、**幂等**(重复 = 再 UPDATE 同值)、**只用 MySQL**(两边都有)。
- 不用 redis / mqtt(小项目过重,ops 也没 redis);以后要秒级再在 outbox 之上加 redis 唤醒,outbox 仍是真相源。
- outbox 可定期清理"所有消费者都已越过"的旧行。
- 与已删的"5 分钟全量同步"不同:这是 append 表的增量拉(通常 0 行),开销极小。

### 11.2 表结构改动汇总(中等偏小)
- **ops `tickets`**:加快照列(见 §3,~10 列,**纯加列、不破坏旧数据**)。
- **ops**:存 `last_seq` 的小处(1 行 kv 表 / 一个 setting)。
- **soyoo**:加 `ops_change_outbox` 表。
- **一次性 backfill**:drop 本地 people/projects 等表**之前**,用它们把旧工单的新快照列填上(否则旧工单显示空)。
- **删表**(backfill + 代码切换后):people / projects / tenants / tags / project_team / project_member_tags + ops_settings + ops_sync_logs。

### 11.3 token:复用 soyoo 登录 token
ops 登录走 soyoo、已持有用户的 soyoo token → 用户级调用**透传**即可,不另设共享密钥。前提:ops 会话里保存该 token 以便转发(需确认现状)。outbox 那条**系统级拉取**(非用户)用服务级 token 或限本机访问。

### 11.4 标签:工单不显示 soyoo 标签
- 工单显示 = ops 自己的「环节名」+「负责人(人名)」,**不直接显示 soyoo 标签**;标签只用于"环节 ↔ 负责人"匹配,**按 tag id(稳定)**。
- 标签改名(美术→UI):环节配置页"选标签"列表实时显示新名;负责人匹配按 id 不受影响;**工单显示不受影响**。
- 若将来要在工单上展示某人角色标签 → 按"改名刷快照"处理(再议)。
- 区分:标签**改名**= 改信息(刷新有显示的地方);给某人**换标签分配**= 类似换人(旧工单不动,只影响以后匹配)。
