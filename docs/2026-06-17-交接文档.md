### 环境变量(关键,容易踩)
- `runtime.mjs` 只加载 **`.env.${APP_ENV}`**(默认 `dev` → **`.env.dev`**)。**没有 `.env` 回退了**(`.env`/`.env.example` 已删)。所以本地配置改 **`.env.dev`**。
- `apps/server/.env`(只有 `DATABASE_URL`)是 **Prisma CLI 专用**(`db pull`/`generate`),跟运行时无关。
- `.env.dev` 里有:本地 MySQL(root/root)、`PORT`、**`COMPANYPLAN_OPS_BASE_URL=http://127.0.0.1:5002`(本地 soyoo)**、OSS_*。
- `.env.prod` 里 `COMPANYPLAN_OPS_BASE_URL=https://helperapi.soyootech.com`(线上 soyoo)。

### ⚠️ 最大的坑:dev 同步的是「本地 soyoo」
- ops 的人员/项目/标签都从 soyoo 的 `/ops/*` 接口同步来。**dev 配的是本地 soyoo `127.0.0.1:5002`**。
- 用户在**本地 soyoo**「项目人员分配」加的成员,要 dev 同步才能用;在线上 soyoo 加的本地拉不到(反之亦然)。排查"建单没有负责人"先确认**同步源对不对 + 同步跑过没**。

### OSS(阿里云)
- bucket `soyoo-ops-materials` @ `oss-cn-beijing`,目录 `ops-materials/{项目id}/{uuid}.{ext}`。
- 凭证在 `.env.dev` / `.env.prod` 的 `OSS_ACCESS_KEY_ID/SECRET`(**明文真实密钥,敏感;分享/上库前应轮换**)。

  - **给 tickets 等加字段、且要被 Prisma 读写时**:① `db/company-plan-store.mjs` 的 `migrateSchema()` 加 `ensureColumn` ② 跑 `cd apps/server && npx --no-install prisma db pull && npx --no-install prisma generate`(否则 Prisma 不认新字段会报错)。
  - 只用 mysql2 的字段(如 `ops_settings`/`ops_sync_logs`)不需要 db pull。

## 3. 数据模型(MySQL,库 companyplan)
| 表 | 用途 / 关键点 |
|---|---|
| `people` | 用户(同步自 soyoo /ops/users)。`discipline`/`title` 已弃用(改用 project_member_tags 解析负责人)。`disabled_at` 软禁用列(**目前同步不会回填,见 TODO**) |
| `projects` | 项目(同步)。`tenant_id`/`planner_name`/`developer_name` 三列 |
| `tenants` | 客户(同步) |
| `tags` | soyoo 标签(同步;只更新 name/color) |
| `project_member_tags`(project_id, person_id, tag_id) | **核心**:项目-成员-标签。建单按"项目+环节"找负责人靠它。同步时按项目整体 DELETE+INSERT |
| `ops_segments`(id, name, default_delivery_hours, risk_warning_hours, sort_order) | **环节**(ops 自定义分类),交付/阈值配置 |
| `ops_segment_tags`(segment_id, tag_id) | **环节 ↔ soyoo 标签 绑定**(程序 ← cocos开发/unity开发…) |
| `tickets` | 提单。`id`=uuid;`segment_id`=**关联环节(显示按 id 取当前名)**,`discipline`=环节名快照(历史回退,非 FK);`due_in_hours`/`risk_warning_hours`=**建单时从环节快照**;`content_html` MEDIUMTEXT=富文本正文;`summary`=纯文本摘要。**`need_type`(任务类别):建单无单独入口,后端默认 `need_type = needType||环节名` → 与「环节」重复,前端已隐藏不展示;字段保留在库/接口(`mapTicket` 仍返回 needType),将来要做成真正「需求类型」再启用** |
| `ticket_events` | 流转记录(建单/状态变更/改说明) |
| `ops_settings`(setting_key, setting_value) | 键值配置(`sync_interval_minutes`/`sync_enabled`) |
| `ops_sync_logs` | 每次同步一条(触发方/操作人/结果/数量/耗时/错误) |
| `attachments` | 附件表(**目前没用上**,见 TODO) |

### 几个"快照 vs 关联"要点(重要)
- 工单的**环节(discipline)、交付时长、预警阈值** 都是**建单时的快照**,不是按 id 实时关联环节表。
- **环节改名**:工单存 `segment_id`,列表显示按 id 取**当前**环节名(改名即时反映);`discipline` 仅作历史快照回退(PUT 里仍有 `updateMany` 同步快照,冗余但无害)。**注**:交付时长/预警阈值仍是建单快照,不随环节配置回填老单。

## 8. 常见坑速查
- 改 `.env.dev` 不生效 → 后端要**手动重启**(watch 不监听 .env)。
- 建单没负责人/环节 → 1) 同步源对不对(dev=本地 soyoo)2) 同步跑过没(去同步管理点一次)3) 该成员在 soyoo 有没有**标签** 4) 该标签在「标签绑定」**绑没绑环节**。
- `mysql -e` 中文乱码/不匹配 → 加 `--default-character-set=utf8mb4`。
- Prisma 报"unknown field" → 加列后忘了 `db pull && generate`。
- 列表崩 `.map of undefined` → 后端没返该数组(旧实例);前端已加 `?? []` 兜底,后端重启确保返回。
- 前端看不到改动 → **硬刷新**;新装依赖偶尔要重启 vite。
- **ops 后端改了不生效** → 当心同时跑了两个实例:`npm run dev:api`(**不监听文件**)和 `pnpm dev:api:watch`。`lsof -nP -i :4174` 看是谁在监听,只留一个(推荐 `dev:api:watch`)。另:`prisma generate` 写的是 `node_modules`,**watch 不会因此重启**,改完 schema 要手动 touch 一个 `.mjs` 或重启在跑的进程。
